ABI：应用程序二进制接口
深度技术解析

从基础概念到 C ABI 的事实标准地位，深入剖析 Rust、C++ 的 ABI 问题，覆盖工程实践与跨语言互操作全链路。

📅 2026 年 6 月 📑 覆盖：C · C++ · Rust · Go · Swift · Python 🔧 System V AMD64 · Windows x64 · Itanium ABI
TL;DR — 三句话摘要

ABI 是编译后二进制代码层面的契约，规定调用约定、数据布局、寄存器使用、名称修饰等，决定两个独立编译的二进制模块能否正确互操作；API 是源码层面的契约，两者有本质区别。

C ABI 因其简单、无修饰、与系统调用绑定，成为所有语言跨语言互操作（FFI）的事实标准"通用语"——任何语言要与操作系统交互，最终都要经过 C ABI。

Rust 故意不稳定化原生 ABI（保留布局优化自由度），通过 extern "C" + #[repr(C)] 互操作；C++ 因 vtable/RTTI/异常/名称修饰复杂性，GCC/Clang（Itanium ABI）与 MSVC ABI 互不兼容，WG21 2020 年投票决定不破坏现有 ABI。

第一章

什么是 ABI

ABI（Application Binary Interface，应用程序二进制接口）是软件在"进程内机器码访问"层面暴露的接口——它是两个独立编译的二进制程序模块之间的契约，规定了数据如何在内存中组织、函数如何被调用、系统如何管理资源。

ABI 与 API 的本质区别

这是理解 ABI 的第一步，也是最容易混淆的地方：

图 1 · API vs ABI — 左侧是人类书写的源码契约，经编译器翻译后形成右侧的二进制契约。API 变更只需重新编译；ABI 破坏则可能引发运行时崩溃而无任何编译期警告。

ABI 包含哪些内容

一个完整的 ABI 覆盖以下所有维度：

维度具体内容破坏后果
调用约定参数顺序、寄存器/栈传参、返回值位置、栈清理责任参数被错误读取，返回值丢失
数据类型布局标量大小、对齐、填充（padding）、位域规则结构体字段偏移错误，数据损坏
名称修饰源码符号→链接器符号名的映射规则链接时找不到符号（undefined reference）
对象模型C++ vtable 布局、RTTI 结构、异常处理表虚函数派发错误，类型转换失败
寄存器保存约定哪些寄存器由调用者/被调用者保存寄存器被意外覆盖，栈损坏
💡 关键认知

ABI 稳定性意味着：不同版本编译的二进制无需重新编译即可正确协同工作。ABI 破坏（ABI breakage）最危险之处在于，它在链接期可能完全没有报错，而是在运行时以段错误（segfault）或静默数据损坏的形式爆发。

第二章

为什么 C ABI 是事实标准

"C 类型是通用语，因为 C ABI 是通用语。"任何语言要打开文件、创建进程、分配内存，最终都必须通过平台的 C ABI 到达内核。每个平台上，C 编译器的调用约定成为该平台的标准 ABI，所有其他语言需要互操作时都以此为目标。

图 2 · C ABI 作为通用桥梁 — 所有高层语言通过各自的 FFI 降级到 C ABI，再经 libc/syscall 到达硬件。C ABI 是这条链路上唯一通用的"通用语"。

System V AMD64 vs Windows x64：两套不兼容的 C 调用约定

即使是 C ABI，在不同平台上也有差异。x86-64 上存在两套主流调用约定，它们互不兼容：

图 3 · System V AMD64 vs Windows x64 — SysV 用 6 个寄存器传整数参数并设有 128 字节红区；Windows 只用 4 个并要求调用者强制预留 32 字节影子空间。即使都是 x86-64，两套 ABI 也互不兼容。

ARM AAPCS64 与其他架构

ARM AArch64 遵循 AAPCS64 规范：X0–X7 传整数参数和返回值，V0–V7 传浮点；X19–X28 为被调用者保存寄存器。RISC-V、PowerPC 等也各有处理器补充（processor supplement），均以 C ABI 为中心构建。

第三章

Rust ABI 问题

Rust 的原生 ABI（extern "Rust"）故意设计为不稳定——这是一个有意的工程权衡，而非缺陷。理解这一点，是掌握 Rust 互操作设计的关键。

图 4 · Rust ABI 全景 — 顶部是"故意不稳定"的原因（保留编译器优化自由度），三条解决路径，底部是 #[repr] 注解族和 v0 符号修饰规则。

通过 extern "C" 与 C 互操作

// 导出可被 C 调用的函数
#[no_mangle]
pub unsafe extern "C" fn rsvg_handle_new_from_file(
    filename: *const libc::c_char,
    error: *mut *mut glib_sys::GError,
) -> *const RsvgHandle { /* ... */ }

// Option<extern "C" fn> 是可空函数指针的正确表达
// None = null，无需额外判别位（空指针优化）
let callback: Option<extern "C" fn(c_int) -> c_int> = None;
⚠ 常见陷阱

用空枚举（uninhabited enum）作 FFI 类型极度危险：编译器依赖空枚举不可居留的性质做激进优化，处理 &Empty 可能触发未定义行为。历史上 #[repr(C)] 枚举在 System V ABI 上按值传递有 bug（rust-lang/rust #68190），曾导致段错误。

v0 名称修饰方案

RFC 2603 设计的 v0 方案以 _R 开头，以可逆方式编码完整类型信息（包括泛型实参），支持 backref 压缩，并于 2025 年 11 月成为 nightly 默认方案。注意：v0 明确不是 Rust 稳定 ABI 规范，仅是符号格式标准。

语言 ABI 稳定性横向对比

语言原生 ABI 稳定性互操作方式代表方案
C稳定直接System V / Win64
C++平台内稳定extern "C" 包装Itanium ABI（GCC/Clang）

第四章

C++ ABI 问题

C++ ABI 是所有主流语言中最复杂的，因为它必须同时解决命名空间/重载/模板的名称修饰、虚函数派发的 vtable、运行时类型识别（RTTI）和异常处理——而 C++ 标准对这些全部未作规定。

图 5 · C++ ABI 核心问题 — 左侧：Itanium ABI 的 vtable 内存布局及 GCC/MSVC 名称修饰的不兼容性。右侧：GCC 5 引入的 libstdc++ 双 ABI 历史事件，及 WG21 2020 年"不破坏 ABI"的关键决策。

Itanium vs MSVC 名称修饰规则

// 同一个函数，两套编译器产生完全不同的符号名：

// GCC / Clang（Itanium ABI）：
int add(int, int)  →  _Z3addii
  // _Z = Itanium 前缀，3 = 名称长度，add = 函数名，ii = int,int

// MSVC：
int add(int, int)  →  ?add@@YAHHH@Z
  // ? = 修饰开始，@@ = 全局作用域，Y = 普通函数
  // A = __cdecl，H = int 返回，HH = int,int 参数，@Z = 结尾

// C++ 成员函数（Itanium）：
Foo::bar(int)  →  _ZN3Foo3barEi
  // N = 嵌套名，3Foo = 类名，3bar = 方法名，E = 结束，i = int
🔥 历史教训：GCC 5 双 ABI 事件

GCC 5.1（2015）将 std::string 从 COW 实现迁移到 C++11 要求的 SSO 实现，布局完全不同。新实现放入内联命名空间 std::__cxx11，由宏 _GLIBCXX_USE_CXX11_ABI 控制。混链不同值编译的对象会触发链接错误，遇到 undefined reference to 'std::__cxx11::...' 即为此问题。至今仍是 Linux 生态 C++ 库最常见的兼容性问题之一。

WG21 的"三选二"困境

2020 年布拉格会议，C++ 标准委员会就 ABI 破坏进行系列投票，最终选择不破坏 ABI。Bryce Adelstein Lelbach 总结了这一永恒困境：

"Performance / ABI Stability / Ability to Change — You can pick two, choose wisely."

许多影响深远的库改进（如更好的哈希函数、优化的容器布局）因 ABI 顾虑在提案阶段就被过滤掉，这是 C++ 生态为稳定性付出的长期代价。

第五章

其他语言的 ABI

Swift：Apple 平台的成功案例

Swift 5.0（2019）在 Apple 平台实现了 ABI 稳定，Swift 运行时和标准库随 OS 发布，应用无需再嵌入它们（此前每个应用约增加 5 MB）。关键机制：

  • @frozen（Swift 5.1）— 承诺结构体不增删重排字段，枚举不增删 case；是 ABI 稳定的核心承诺
  • @inlinable（Swift 4.2）— 允许调用方内联实现，以 ABI 灵活性换性能
  • .swiftinterface 模块接口文件（Swift 5.1）— 跨编译器版本的二进制框架分发
ℹ️ 注意

Swift ABI 稳定仅在 Apple 平台（macOS/iOS/watchOS/tvOS）上有效。Linux 和 Windows 上 Swift 的 ABI 尚未稳定，Swift Core Team 称随这些平台成熟会评估。

Go：双 ABI 设计

Go 运行时维护两套 ABI 并通过透明包装器互相调用：

ABI特性用途
ABI0栈式调用约定（Plan 9 风格）汇编代码使用，跨语言边界的稳定契约
ABIInternal寄存器式（Go 1.17 引入），比 ABI0 快约 40%纯 Go 函数内部调用，随版本演进

Go 刻意使用自有 ABI 而非平台 ABI，因为 goroutine 栈动态伸缩的需求（运行时可动态调整栈大小），以及简化 GC 追踪和 panic 展开的工程需求。

Python：Stable ABI / abi3

Python 3.2 引入 Limited API（Py_LIMITED_API 宏）及对应的 Stable ABI（abi3）——只用 Limited API 编译的扩展一次构建即可在多个 Python 3.x 版本上运行，无需为每个版本重新编译。核心价值：O(1) 构建——旧 wheel 在新 Python 上仍可用。

# setup.py 指定 abi3，扩展文件名为 myext.abi3.so
from setuptools import setup, Extension
setup(ext_modules=[Extension(
    "myext",
    ["myext.c"],
    py_limited_api=True,    # 启用 Limited API
    define_macros=[("Py_LIMITED_API", "0x030a0000")],  # 最低 3.10
)])

第六章

工程实践

图 6 · ABI 工程实践 — SONAME 三层命名机制（真实文件→DT_SONAME→链接名），ELF 符号版本化（让旧程序在新系统上无需重编译的核心机制），以及 libabigail 的 CI 检测流程。

ELF 符号版本化实战

# 强制绑定到旧版符号（避免依赖新版 glibc）
__asm__(".symver clock_gettime,clock_gettime@GLIBC_2.17");

# 版本脚本（version script）
MYLIB_2.0 {
  global:
    my_func_v2;     # 新版本导出
  local:
    *;              # 隐藏所有内部符号
};

MYLIB_1.0 {
  global:
    my_func;        # 旧版本保留
} MYLIB_2.0;        # 继承关系

# 检查二进制依赖的最低 glibc 版本
objdump -T libfoo.so | grep GLIBC

ABI 设计最佳实践

接口设计

公共接口用 extern "C" 暴露。不在 ABI 边界暴露 C++ 模板、STL 容器、Rust 原生类型或异常。用 pimpl 惯用法隐藏实现细节。

版本管理

ABI 破坏时递增 SONAME 主版本号。用版本脚本控制导出面、隐藏内部符号。在结构体尾部添加字段并预留保留字段。

CI 检测

将 abidiff（libabigail）纳入 CI，对照上一发布版检测。返回码 bit4=1 设为构建失败，bit3=1 设为人工评审。

Rust 场景

默认静态链接。只在需要运行时插件/动态加载时才引入 abi_stable 或 stabby，并接受"加载时校验布局"的代价。

C++ 场景

在同一发行链内统一 GCC/libstdc++ 主版本，明确并固定 _GLIBCXX_USE_CXX11_ABI 取值并在文档中声明。

Python 场景

用 Limited API 编译 abi3 wheel，用 abi3audit 验证标记与实际一致，避免"标记兼容、实际依赖新符号"的隐患。

💡 核心原则

跨语言/跨编译器互操作，一律以 C ABI 为边界。若头文件中出现 std::string、std::vector、模板或 inline 函数作为导出接口，就已经把自己绑定到特定编译器/标准库 ABI。两个不同 rustc 构建的 cdylib 之间，不要传递任何非 #[repr(C)] 类型。
